# Toyota Connected Services Integration Plan

## Overview
This document outlines a plan to integrate European Toyota Connected Services into Fuelog. The integration will allow users to link their Toyota accounts to automatically sync vehicle details, mileage, and fuel events.

## Analysis of the Integration
Based on an analysis of the unofficial `pytoyoda` Python client, the Toyota Connected Services API in Europe utilizes an OAuth 2.0 flow integrated with a ForgeRock OpenAM identity provider (`b2c-login.toyota-europe.com`).

The authentication process requires:
1. Submitting a username and password to an authentication endpoint.
2. Generating a `tokenId` cookie.
3. Exchanging this cookie for an OAuth 2.0 authorization code.
4. Exchanging the authorization code for an `access_token`, `refresh_token`, and `id_token`.

Data retrieval requires these tokens, specifically calling endpoints on `https://ctpa-oneapi.tceu-ctp-prd.toyotaconnectedeurope.io` with several specific custom headers (e.g., `x-api-key`, `x-guid`, `x-channel`, `x-brand`, etc.).

## Recommended Architecture

### 1. Backend Service (Firebase Cloud Functions)
Since the Toyota API relies on a complex authentication flow that might change and involves secrets, we should handle this via a backend service.

* **Authentication Endpoint (`/api/toyota/auth`)**: A secure endpoint where a user submits their Toyota username and password. The backend performs the login flow and extracts the `refresh_token` and `id_token`. We do *not* store the password. We only store the `refresh_token` securely in Firestore against the user's profile or vehicle document.
* **Sync Function (`/api/toyota/sync` or Scheduled Cron Task)**: A function that runs periodically (e.g., nightly) or manually triggered by the user. It uses the stored `refresh_token` to get a fresh `access_token`.
* **Data Retrieval**: Once authenticated, the function calls the `/v1/trips` or `/v3/telemetry` and `/v1/global/remote/electric/status` endpoints to retrieve the latest mileage and fuel level.
* **Fuel Logging Logic**: By comparing the current fuel level and mileage with the previous sync, the system can detect if a refueling event occurred (a significant positive delta in fuel level). When a refueling event is detected, it automatically creates a new record in the `fuelLogs` collection.

### 2. Database Changes (Firestore)
* **`vehicles` Collection**: Add fields to store the Toyota linkage state to `Vehicle` and `FuelLogData` interfaces in `src/utils/types.ts`.
    * `toyotaRefreshToken` (string): The OAuth refresh token (encrypted via KMS/secret key).
    * `toyotaUuid` (string): The UUID extracted from the `id_token`.
    * `toyotaVin` (string): The linked Vehicle Identification Number.
    * `toyotaLastSync` (Timestamp): Timestamp of the last successful sync.
    * `toyotaLastFuelLevel` (number): The last known fuel level (%) to detect refueling events.
    * `toyotaLastMileage` (number): The last known mileage (km) to calculate distance covered.
* **`fuelLogs` Collection**: Add an optional field `isToyotaSynced` (boolean) to `FuelLogData` to mark logs that were automatically generated by the integration.

### 3. Frontend Integration (UI)
* **Profile Page (`src/pages/ProfilePage.tsx`)**:
  * Add a "Connect Toyota Account" component beneath or above the "API Access Section".
  * This will present a form for their Toyota username and password (with clear disclaimers that credentials are only used once to obtain a token and are not stored).
  * Display a list of linked vehicles with their Toyota VIN and sync status (last sync time).
* **Vehicle Cards**: Update the vehicle cards in `ProfilePage.tsx` to display a small "Toyota Connected" badge if the vehicle has `toyotaVin` set.
* **Sync Status**: Add a manual "Sync Now" button on the linked vehicle cards to trigger a manual update of fuel logs from the backend.

### Security Considerations
* **Never store passwords**: The authentication flow must only happen once in memory. Only the resulting `refresh_token` should be persisted.
* **Encryption at Rest**: Although `refresh_token`s are bearer tokens that eventually expire or are revoked, they should be treated as highly sensitive. Consider using Google Cloud KMS or a secret key to symmetrically encrypt these tokens before storing them in Firestore.
* **API Key Management**: The static `x-api-key` used by the client needs to be stored as a backend environment variable.

## Next Steps for Implementation
1. **Develop a TypeScript library**: Port the essential parts of `pytoyoda` (authentication, telemetry fetching, token refresh) into a TypeScript module in the backend (`functions/src/toyota-api/`).
2. **Implement Backend Endpoints**: Create the Firebase Functions for login and syncing.
3. **Database Schema Update**: Modify security rules and typings to support the new Toyota connection fields.
4. **Build Frontend Components**: Develop the authentication UI and linking workflow.
5. **Testing**: Add extensive unit tests for the TypeScript Toyota API client to ensure resilience against API changes.
